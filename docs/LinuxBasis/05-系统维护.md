# 核心：`chroot`命令
`chroot`将一个目录转化为新的根目录，使系统在该目录下运行，相当于另外开了一个系统。这个新开的系统可以是待修复的系统，也可以是一个用于软件测试的“干净”系统。本文档主要涉及到前者，系统维护。

假设待修复的系统为硬盘上的 Linux1，记为`/dev/nvme0n1p11`。如何使用`chroot`新开一个系统？

1. 使用 Live USB 启动 Linux2，是一个临时系统，用于修复 Linux1。Linux2好比医生，Linux1好比病人。
2. 在 Linux2 上挂载 Linux1 的根目录到 `/mnt/linux1`：
   ```bash
   sudo mount /dev/nvme0n1p11 /mnt/linux1
   ```
3. 还要挂载其他必要的目录，如 `/dev`、`/proc`、`/sys` 等：
   ```bash
   sudo mount --bind /dev /mnt/linux1/dev
   sudo mount --bind /proc /mnt/linux1/proc
   sudo mount --bind /sys /mnt/linux1/sys
   ```
   > 有的小伙伴可能会问，`/dev/nvme0n1p11`不是有`dev`,`proc`,`sys`等目录吗，为什么还要挂载它们呢？
      将 `/dev/nvme0n1p11` 挂载到 `/mnt/linux1` 时，确实得到了 Linux1 中的所有目录结构，包括 `/dev`,`/proc`,`/sys` 等目录。但这些只是空目录，不包含当前运行系统的实际设备文件和内核接口。还需要让被 `chroot` 的Linux1 能够：
         - 访问硬件设备（通过 /dev）
         - 与内核通信（通过 /proc 和 /sys）
         - 管理进程信息
         - 处理网络连接等
      明白了吗？

   以上一个设备、三个目录只是必要的，有时还会附加其他目录，例如`boot/efi`，详情参见[修复引导项](#修复引导项)。
4. 现在，我们可以 `chroot` 到 `/mnt/linux1` 了。
   ```bash
   sudo chroot /mnt/linux1
   ```
   这时，开启了一个子bash，提示词会变成`root@C7VF:/#`，认为`/`是`/mnt/linux1`，在这个子bash下执行的命令都来自 Linux1中的可执行文件，且对 Linux1 生效。这些都表示你现在在 Linux1 系统中的环境中。
5. 修复完毕后，按一下`Ctrl+D`或执行`exit`，即可退出 Linux1，回到 Linux2。
6. 最后，卸载挂载的目录：
   ```bash
   sudo umount /mnt/linux1/dev
   sudo umount /mnt/linux1/proc
   sudo umount /mnt/linux1/sys
   sudo umount /mnt/linux1
   ```

以上是`chroot`维护系统的基本、通用的流程，如果还不理解，可以参考下面的示例。

## 示例一：修复NVIDIA显卡驱动
我的电脑是Windows和Ubuntu双系统型。在Ubuntu中对NVIDIA显卡进行了一些操作后，Ubuntu的各种模块都不能加载，有显卡、网卡、有线网络、蓝牙、屏幕亮度调节等，但在Windows里面这些都可以正常使用。所以这属于软件而非硬件问题，且由于Ubuntu因无法连接网络而安装驱动，故欲使用 Live USB 进入一个临时Ubuntu系统，连接网络后进行驱动的重新安装，从而实现修复。

假设现在已经进入 Live USB 系统，且连接了网络，已知硬盘上的Ubuntu系统为`/dev/nvme0n1p11`，EFI分区（引导项在全部在这里）为`/dev/nvme0n1p1`。之后的步骤如下：

1. 挂载，chroot到硬盘系统
   ```bash
   # 创建挂载点目录
   sudo mkdir /mnt/ubuntu
   
   # 挂载根分区
   sudo mount /dev/nvme0n1p11 /mnt/ubuntu  # 请将 /dev/nvme0n1p11 替换为你实际的根分区
   
   # 挂载虚拟文件系统，这是chroot所必需的
   sudo mount --bind /dev /mnt/ubuntu/dev 
   sudo mount --bind /dev/pts /mnt/ubuntu/dev/pts
   sudo mount --bind /proc /mnt/ubuntu/proc
   sudo mount --bind /sys /mnt/ubuntu/sys
   sudo mount --bind /run /mnt/ubuntu/run # 作用：挂载运行时目录，使被chroot的系统能够访问系统运行时的必要文件和目录。

   sudo chroot /mnt/ubuntu
   ```
3. 针对硬盘上Ubuntu系统的操作
   ```bash
   apt update
   apt upgrade

   # 清除所有已安装的NVIDIA驱动包
   apt remove --purge nvidia-*  
   # 自动安装推荐的NVIDIA驱动
   ubuntu-drivers autoinstall 

   # 更新初始化内存盘（initramfs）。
   # 这一步至关重要，它确保新安装的驱动和内核模块在下次启动时能被正确加载。
   update-initramfs -u -k all 

   # 重新配置GRUB（可选，但推荐）
   update-grub
   ```
4. 退出chroot环境，卸载所有挂载的分区，重启系统
   ```bash
   exit

   sudo umount /mnt/ubuntu/run
   sudo umount /mnt/ubuntu/sys
   sudo umount /mnt/ubuntu/proc
   sudo umount /mnt/ubuntu/dev/pts
   sudo umount /mnt/ubuntu/dev
   sudo umount /mnt/ubuntu # 如果直接进行这一步，会提示目标忙

   sudo reboot
   ```
5. 重启进入系统后，各个模块应该能正常使用了。

## 示例二：修复引导项 {#修复引导项}
参考：https://gparted.org/display-doc.php?name=help-manual&lang=C#gparted-fix-grub-boot-problem

```bash
# 创建挂载点目录
sudo mkdir /mnt/ubuntu

# 挂载根分区
sudo mount /dev/nvme0n1p11 /mnt/ubuntu  # 请将 /dev/nvme0n1p11 替换为你实际的根分区

# # 挂载Boot/EFI分区
sudo mount /dev/nvme0n1p1 /mnt/ubuntu/boot/efi  # 请替换为你实际的EFI分区

# 挂载虚拟文件系统，这是chroot所必需的
sudo mount --bind /dev /mnt/ubuntu/dev 
sudo mount --bind /proc /mnt/ubuntu/proc
sudo mount --bind /sys /mnt/ubuntu/sys
sudo chroot /mnt/ubuntu

# 修复引导项
grub2-install
```

### 坑点一：挂载点要正确
我掉过这个坑，导致最后重启时除了BIOS外什么系统都进不去！！那就是网站提到的下面一句：
```bash
mount /dev/sda3 /tmp/mydir/boot
```
我的引导项在`/dev/nvme0n1p1`里，但它挂载的是`/boot/efi`而不是`/boot`，我忘记了这个差异，按网站的挂载到`/tmp/mydir/boot`,导致后续`grub-install`时使引导项的文件结构错误，重启时除了BIOS外什么都进不去。对于这一步，我应该执行的是
```bash
mount /dev/nvme0n1p1 /tmp/mydir/boot/efi
```
是该好好了解`/boot`目录了，参见[04-多重引导.md](04-多重引导.md#boot)

### 不算太坑的点二：权限问题
`mount`需要root权限，需要`sudo`。

### 坑点三：是不是GRUB boot loader
网页本身也说了，这段参考成立的前提是"If your computer uses the GRUB boot loader"。怎么知道自己的电脑使用的是不是**GRUB boot loader** 呢？（而不是像 systemd-boot、rEFInd 或 Windows Boot Manager 这样的其他引导器）可以根据以下几种方法判断：

1. **看开机启动界面**
    开机时是否出现如下界面：
    ```
    GNU GRUB version 2.06
    -------------------------------------
    * Ubuntu
      Advanced options for Ubuntu
      Memory test
    -------------------------------------
    ```
    或者菜单底部写着：
    ```
    Minimal BASH-like line editing is supported.
    grub>
    ```
    👉 有这个界面 = 使用GRUB。

    如果没有这个界面呢？也不一定不是GRUB引导器，只是可能设置成启动时不出现GRUB界面罢了，要解决不出现GRUB界面的问题，请按如下方法做：
    ```bash
    nano /etc/default/grub
    # 找到这行：
    # GRUB_TIMEOUT_STYLE=hidden
    # 改为：
    # GRUB_TIMEOUT_STYLE=menu
    # 设置GRUB界面倒计时时间（秒）
    # GRUB_TIMEOUT=10
    # 取消注释下面这一行：
    # GRUB_DISABLE_OS_PROBER=false

    update-grub
    ```

2. **进入系统后执行命令检查**
    ```bash
    ls /boot/grub
    ```
    如果有输出，说明 `/boot/grub` 目录存在，说明系统安装了 GRUB 引导器。再进一步确认版本：
    ```bash
    grub-install --version
    ```
    输出例子：
    ```
    grub-install (GRUB) 2.06
    ```
3. **查看EFI系统分区下的启动器**
    ```bash
    sudo ls /boot/efi/EFI/
    ```
    输出可能是：
    ```
    ubuntu  fedora  BOOT  Microsoft
    ```
    然后：
    ```bash
    sudo ls /boot/efi/EFI/ubuntu
    ```
    可能看到：
    ```
    grubx64.efi  shimx64.efi
    ```
    说明这个系统用的是 **GRUB**（`grubx64.efi`）作为UEFI启动器。
4. **使用 `efibootmgr` 查看当前使用的启动项**
    ```bash
    sudo efibootmgr
    ```
    输出例子：
    ```
    BootCurrent: 0001
    BootOrder: 0001,0002
    Boot0001* ubuntu
    Boot0002* Windows Boot Manager
    ```
    结合上一个方法看到的是 `grubx64.efi`，就能确认：

    👉 当前是通过GRUB引导 Ubuntu 的。

5. **（双保险）检查内核命令行来源**
    ```bash
    cat /proc/cmdline
    ```
    如果你看到的内容类似于：
    ```
    BOOT_IMAGE=/boot/vmlinuz-6.8.0 root=UUID=... ro quiet splash
    ```
    这通常也是 GRUB 传递的启动参数（不过这一点也可能是 systemd-boot）。

6. **BIOS/UEFI里查看“Boot Options”名称**

    重启进入 BIOS 设置界面，在 "Boot Order" 或 "Boot Menu" 中：

    * 如果看到类似“Ubuntu”或“Debian”，通常是 GRUB 创建的启动项。
    * 如果看到“rEFInd”或“systemd-boot”，说明你用的是其他引导器。
